# AI 视觉对话助手设计文档

## 1. 项目定位

本项目面向学习辅导场景：学生把题目放在摄像头前，通过语音提问，AI 根据画面和语音进行引导式回答。目标不是替学生直接解题，而是像同学或助教一样追问知识点、提示下一步、帮助学生自己想明白。

核心交互链路为：

```text
摄像头预览/框选题目 -> 端侧 VAD 检测说话 -> 截取当前画面并做视觉转写
-> ASR 识别语音 -> DeepSeek 结合语音文本、图片转写和上下文流式回答
-> TTS 分句合成播放 -> 继续监听
```

其中最重要的创新点是“视觉区域框选”：用户可以只框选题目或答题区域，系统不必上传整张视频截图，既节省视觉 token，又减少桌面、背景、手部遮挡等干扰元素。

## 3. 关键设计

### 3.1 Web 端链路

1. `useCamera` 打开摄像头并提供截图能力。
2. `CameraView` 负责视频预览和框选区域。
3. `useVAD` 在浏览器端判断语音开始/结束。
4. 语音开始时截取当前视频帧；如果存在框选区域，就调用 `cropFromVideo` 只裁剪该区域。
5. 语音开始后立即并行调用 `qwen-vl-plus` 做图片客观转写，只提取题目、公式、布局和可见文字。
6. 语音结束后，`useASR` 将音频编码成 16kHz WAV 并调用 `qwen3-asr-flash`。
7. `useDialogue` 内部调用 DeepSeek 流式生成最终学习引导，输入包含语音文本、图片转写和历史上下文。
8. `useTTS` 监听流式文本，遇到句末标点就分句调用 `qwen3-tts-flash`，边生成边播放。

### 3.2 Android 端链路

Android 版本使用原生 Camera、TextureView 和 AudioRecord：

- `SelectionOverlayView` 提供触摸框选。
- `AudioVadRecorder` 用 RMS 阈值、起始确认、静音结束和最长语音时长控制录音片段。
- `DashScopeClient` 负责 ASR、视觉转写和 TTS。
- `DeepSeekClient` 负责最终学习助手回答，输入包含语音文本、历史上下文和可选图片转写。
- TTS 播放期间暂停麦克风监听，避免 AI 自己的声音被再次识别。

## 4. 成本控制策略

| 策略 | 预期作用 | 实际采用情况 |
| --- | --- | --- |
| 不上传连续视频，只在用户说话时截取单帧 | 避免视频流高额 token 和带宽成本 | 已采用。每轮对话只处理语音开始时的一张截图。 |
| 支持框选题目区域 | 减少图片面积、降低视觉 token、减少无关背景干扰 | 已采用。Web 和 Android 均支持框选裁剪。 |
| 图片压缩和尺寸限制 | 降低上传体积和视觉模型输入成本 | 已采用。Web 压缩到约 720p/0.3MB；Android 限制长边并用 JPEG 质量压缩。 |
| 只在文本意图需要时发送图片 | 普通闲聊/追问不消耗视觉 token | 已考虑，早期 Web 版本采用关键词规则；为保持 Web 和 Android 行为一致，当前版本改为每轮截图后做视觉转写，由最终回答模型判断图片是否相关。 |
| 端侧 VAD 过滤静音 | 不把沉默和环境噪音上传给 ASR | 已采用。Web 使用 VAD 模型，Android 使用轻量 RMS VAD。 |
| 对话历史压缩 | 控制上下文 token 随轮次增长 | 已采用。Web 超过轮数后把旧对话压缩成规则摘要；Android 只取最近消息和摘录。 |
| TTS 音频缓存重播 | 用户重听时不重复调用 TTS | 已采用。AI 回复的音频分句会挂到消息上供重播。 |
| 开发期使用更经济的模型 | 控制调试成本 | 已采用。默认使用 `qwen-vl-plus`、`qwen3-asr-flash`、`qwen3-tts-flash`。 |
| API Key 只在本地代理/构建配置中读取 | 降低密钥泄露导致的额外成本风险 | 已采用。`.env` 和 `local.properties` 被加入忽略。 |
| 服务端配额、缓存和限流 | 防止被恶意调用或异常消耗 | 已考虑，暂未实现。当前项目主要面向本地演示。 |

## 5. 视觉框选的取舍

直接上传整张摄像头截图实现最简单，但对学习场景不够理想：题目通常只占画面一部分，整图里可能有桌面、笔、手、屏幕边框和其他题目。框选方案会增加一点交互成本，但带来三个收益：

- 视觉模型更容易聚焦到本轮问题相关区域。
- 图片尺寸和 token 成本更低。
- 学生可以明确表达“我问的是这里”，减少 AI 误读上下文。

因此当前实现优先保证框选区域的坐标映射准确性。Web 端专门处理了 `object-fit: cover` 下 CSS 坐标到视频原生分辨率的转换，避免裁剪位置偏移。

## 6. 语音交互流畅性设计

- 端侧 VAD 自动切分一句话，减少“按住说话”式操作。
- AI 开始回答前先停止上一轮 TTS，避免多段语音重叠。
- 流式文本先显示，TTS 按句合成，不必等整段回答完成。
- TTS 播放期间暂停/延迟麦克风监听，降低回声导致的误触发。
- 回复限制为最多 3 句话，符合学习辅导场景，也能降低 TTS 等待时间。

## 7. 已知限制与后续优化

- Web 版本依赖 Vite 本地代理，若部署公网需要补一个后端代理服务。
- ASR 是非实时一句话识别，不是逐字实时字幕。
- 当前 Web 与 Android 都会在每轮语音开始时做图片转写，稳定性更高，但普通闲聊场景仍可进一步接入轻量意图判断以降低成本。
- 暂无 token/费用统计面板，无法在 UI 中直观看到每轮节省了多少成本。

## 8. 验证方式

- Web：运行 `npm run build` 验证 TypeScript 和 Vite 构建。
- Android：在 Android Studio 中同步 Gradle 并运行 `app`。
- 人工演示：使用根目录的 `演示视频.mp4` 展示摄像头、框选、语音提问、AI 流式回答和 TTS 播放流程。
